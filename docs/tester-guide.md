# Android Use 测试人员使用手册

这份文档面向测试人员，用来把 Android Use 放进日常测试、Bug 复现、回归验收和证据采集流程。你不需要理解插件内部实现，但需要知道什么时候该用截图、什么时候该用控件树、什么时候该录屏，以及插件更新后怎么重新验证。文档后半部分结合最近一个月的实际使用场景，重点覆盖工单排查、学生端 Android/WebView Bug 复现、版本不匹配、原生/H5 归属判断和证据归档。

## 1. Android Use 能帮测试做什么

Android Use 是 Codex 的本地 Android 控制插件。它通过 adb、UIAutomator、截图、Playwright Android WebView、scrcpy 和可选视觉模型控制已连接的 Android 手机、平板或模拟器。

测试工作里最常用的能力：

- 列出和识别 Android 设备；
- 打开 App、打开 URL 或深链；
- 查看当前页面、截图、抓 UI 控件树；
- 点击文字、点坐标、滑动、输入文本、按返回键或 Home 键；
- 对混合 App / H5 页面优先走 WebView DOM 快路径；
- 录制 MP4，作为缺陷附件；
- 生成 AppShot，把截图、设备状态和控件树一起保存；
- 录制一段可复放 recipe，用于重复回归路径；
- 执行只读 adb shell 命令，辅助定位版本、包名、前台 Activity 和日志线索。

不要把它当成自动化测试平台的完整替代品。它更适合作为测试人员的本地助手：快速进入页面、稳定复现、拿到证据、让 Codex 帮忙执行重复步骤。

## 2. 安全边界

测试人员可以直接让插件做普通操作，例如打开 App、截图、滑动、点击、输入非敏感测试文本、录屏、查看控件树。

下面这些动作必须先得到明确授权，不要只因为屏幕上出现按钮就让插件点击：

- 删除本地或云端数据；
- 发送消息、发帖、上传文件、拨打电话、提交正式表单；
- 购买、支付、订阅、解绑银行卡、身份认证；
- 安装 App、授予危险权限、修改密码、添加账号、修改安全设置；
- 输入或传输密码、验证码、API Key、私有文件、精确位置等敏感信息。

测试账号、验证码、密码和隐私数据不要直接贴进普通对话。确实需要输入时，应使用团队规定的测试账号和脱敏数据。

## 3. 测试前准备清单

电脑侧：

- macOS；
- 已安装 Codex 桌面端；
- 已安装 Android Use 插件；
- `python3`、`adb`、Node.js/npm 可用；
- 推荐安装 `scrcpy`，用于镜像窗口和 MP4 录屏；
- 插件目录通常是 `~/plugins/android-use-plugins`。

设备侧：

- 已安装待测 App；
- 已开启开发者选项；
- 已开启 USB 调试；
- USB 连接后已经在设备弹窗里点过「允许 USB 调试」；
- 如果走无线调试，电脑和设备在同一个 Wi-Fi；
- 设备电量充足，屏幕锁定策略不会频繁打断测试。

测试资料：

- 测试账号；
- 测试环境说明，例如 dev、test、staging、prod；
- App 版本号或安装包来源；
- 目标页面、入口路径、深链或测试用例编号；
- 期望结果和需要采集的证据类型。

## 4. 首次安装和环境校验

如果你拿到的是 `android-use-plugins.zip`，可以直接对 Codex 说：

```text
请帮我解压 android-use-plugins.zip，进入解压后的目录，执行 ./install.sh 安装 Android 插件，然后执行 ./doctor.sh 检查环境。
```

安装完成后，退出并重新打开 Codex。插件列表里应该能看到 `Android`。

如果你自己在终端里操作：

```bash
unzip android-use-plugins.zip
cd android-use
./install.sh
./doctor.sh
```

如果你是从源码仓库安装：

```bash
cd ~/plugins/android-use-plugins
./install.sh
./doctor.sh
```

`./doctor.sh` 是最重要的健康检查。它会检查 adb、Playwright Android WebView 依赖、插件配置、元数据和离线测试。安装、更新、排查问题后都应该先跑它。

校验通过后，在 Codex 里执行：

```text
[@Android] 检查依赖
[@Android] 列出设备，包含型号和屏幕信息
```

如果能看到设备处于 `device` 状态，就可以开始测试。

## 5. 连接设备

### 5.1 USB 连接

最稳定的方式是 USB 数据线。

1. 用数据线连接设备和电脑。
2. 设备弹出 USB 调试授权时点「允许」。
3. 在 Codex 里说：

```text
[@Android] 列出设备
```

常见状态：

- `device`：可用；
- `unauthorized`：设备还没授权电脑，重新插拔 USB 后看设备弹窗；
- `offline`：adb 连接异常，重插线或重启 adb；
- 没有设备：检查是否是数据线、USB 调试是否打开、设备是否选择文件传输模式。

### 5.2 无线调试

Android 11 及以上支持无线调试。首次配对需要设备和电脑在同一个 Wi-Fi。

二维码配对：

```text
[@Android] 创建无线配对二维码
```

用设备在「设置 -> 开发者选项 -> 无线调试」里扫码后，再说：

```text
[@Android] 完成无线配对
```

配对码方式：

```text
[@Android] 无线配对 host=172.27.31.51 pair_port=42123 code=123456
```

后续重连：

```text
[@Android] 无线重连
```

如果已经保存了多台设备：

```text
[@Android] 重连所有已保存的无线设备，并打开 scrcpy
```

### 5.3 多台设备

如果连接了多台设备，测试前先明确序列号：

```text
[@Android] 列出设备，包含型号和屏幕信息
```

后续提示词里带上设备序列号：

```text
[@Android] 使用序列号 ANMB9X5A10G00857 截图
```

也可以让 Codex 先说明它将控制哪台设备，再执行操作。

## 6. Codex 提示词写法

建议用「目标 + 约束 + 证据」的方式写。

普通操作：

```text
[@Android] 打开待测 App，截图并告诉我当前页面标题和前台 Activity。
```

复现 Bug：

```text
[@Android] 按测试用例 TC-102 的路径进入登录页，点击“忘记密码”，输入测试手机号 13800000000，但不要提交验证码。每一步完成后截图。
```

回归验收：

```text
[@Android] 按下面步骤回归：打开 App -> 进入我的页面 -> 点击设置 -> 修改昵称为 test_user_0612 -> 返回我的页面。请保存截图证据，并说明实际结果是否符合预期。
```

只读排查：

```text
[@Android] 只读检查当前 App 包名、版本号、前台 Activity 和 WebView 页面，不要点击提交、删除、支付或发送类按钮。
```

高风险动作前加限制：

```text
[@Android] 进入订单页验证按钮状态，只允许截图和查看控件树，不要点击“支付”“提交订单”“确认退款”。
```

## 7. 日常测试工作流

### 7.1 启动前检查

每次开始一轮测试，建议先跑：

```text
[@Android] 检查依赖
[@Android] 列出设备，包含型号和屏幕信息
[@Android] 唤醒设备并解锁
[@Android] 打开 scrcpy 镜像窗口
```

如果不需要实时镜像，只需要证据时间线：

```text
[@Android] 打开 Android 操作时间线
```

### 7.2 打开 App 和定位页面

知道包名时：

```text
[@Android] 打开包名 com.example.app
```

知道深链时：

```text
[@Android] 用 Android 打开 myapp://test/path?id=123
```

不知道当前状态时：

```text
[@Android] 观察当前屏幕，优先 WebView，必要时带截图。
```

需要明确页面证据：

```text
[@Android] 生成当前 Android AppShot，包含截图、设备状态和控件树。
```

### 7.3 点击、滑动和返回

优先点文字，不要一上来点坐标：

```text
[@Android] 点击文字“登录”
[@Android] 点击文字“设置”
[@Android] 向上滑动一屏
[@Android] 按返回键
```

只有页面没有可识别文字、控件树拿不到、或需要点图标时，再用坐标：

```text
[@Android] 在当前截图中点击右上角更多按钮的位置。
```

### 7.4 输入文本

输入普通测试文本：

```text
[@Android] 在当前输入框输入 test_user_0612
```

清空后输入：

```text
[@Android] 清空当前输入框，然后输入 regression_0612
```

输入后回车：

```text
[@Android] 输入 android use test 并按回车。
```

混合 App / H5 页面中，插件默认会优先走 WebView DOM 输入，速度比键盘逐字输入更快。找不到可调试 WebView 或目标输入框时，会回退到输入法或 adb。

### 7.5 WebView / H5 页面

很多业务 App 是原生壳 + H5 页面。遇到这种页面，先让插件找 WebView：

```text
[@Android] 列出当前可调试 WebView 页面。
```

查看页面信息：

```text
[@Android] 在当前 WebView 执行 document.title 和 location.href，只返回结果。
```

查找页面文本：

```text
[@Android] 观察当前页面，优先 WebView DOM 快照，返回可点击文本和输入框。
```

当页面能被 WebView 识别时，点击文字和输入会更稳定；当页面识别不到 WebView 时，再回退到 UIAutomator、截图或坐标。

## 8. 证据采集规范

### 8.1 最轻量证据：截图

适合记录当前页面状态：

```text
[@Android] 截图并保存。
```

### 8.2 推荐证据：AppShot

适合缺陷复现和回归验收：

```text
[@Android] 生成当前 Android AppShot，包含图片和 UI 控件树。
```

AppShot 通常会保存到：

```text
.screen/appshots/
```

它比普通截图多了设备状态和控件树，便于后续定位前台 Activity、包名、可点击控件和页面结构。

### 8.3 操作时间线

适合演示「每一步怎么走到问题页面」：

```text
[@Android] 打开 Android 操作时间线。
```

之后通过 Android Use 工具执行的动作会被记录成步骤截图。它不是持续视频流，适合轻量复盘。

### 8.4 MP4 录屏

适合动画、闪退、加载、跳转、手势、播放器、输入法等动态问题：

```text
[@Android] 开始录制 Android 屏幕视频，名称用 login-regression-0612。
```

复现完成后：

```text
[@Android] 停止录制视频，并返回本地 MP4 路径。
```

默认输出目录：

```text
.screen/video-recordings/
```

录屏和 recipe 不是一回事。录屏生成 MP4；recipe 记录可复放动作。

## 9. 可复放 recipe

当一个回归路径需要反复跑，可以录制成 recipe。

录制：

```text
[@Android] 开始录制 Android recipe，名称 login-flow。
```

然后执行测试步骤：

```text
[@Android] 打开待测 App。
[@Android] 点击文字“登录”。
[@Android] 输入测试账号 test_user。
[@Android] 点击文字“下一步”。
```

停止并生成 recipe：

```text
[@Android] 停止录制，并把这段 trace 转成 selector 优先的 recipe。
```

下次复放：

```text
[@Android] 复放 login-flow recipe，严格校验关键页面。
```

recipe 会优先使用文本、resource id 等 selector；找不到 selector 时才回退到坐标。适合稳定路径，不适合依赖随机验证码、动态推荐流或强账号状态的路径。

## 10. 缺陷复现模板

可以直接复制下面模板给 Codex：

```text
[@Android] 我要复现一个 Android Bug。

约束：
1. 只使用测试账号和测试环境。
2. 不点击支付、提交订单、删除、发送消息、授权权限等高风险按钮。
3. 每个关键步骤后保存截图。
4. 如果页面无法识别，先返回 AppShot，不要猜测。

设备：
- 优先使用当前已连接设备。

路径：
1. 打开待测 App。
2. 进入「我的」页面。
3. 点击「设置」。
4. 点击「账号与安全」。

期望：
- 页面显示手机号脱敏信息。

请输出：
1. 实际路径是否走通；
2. 每一步证据文件位置；
3. 当前包名、版本号、前台 Activity；
4. 是否复现，若未复现请说明卡在哪一步。
```

## 11. 回归验收模板

```text
[@Android] 请帮我做一次 Android 回归验收。

变更说明：
- 修复昵称编辑后返回个人页未刷新的问题。

测试限制：
- 使用测试账号。
- 只修改昵称为 regression_0612。
- 不改头像、不退出登录、不清理账号数据。

步骤：
1. 打开 App。
2. 进入「我的」页面，记录当前昵称。
3. 进入「编辑资料」。
4. 清空昵称并输入 regression_0612。
5. 点击保存。
6. 返回「我的」页面。
7. 生成 AppShot。

请给出：
1. 实际结果；
2. 是否符合预期；
3. 截图或 AppShot 路径；
4. 如果失败，停在当前页面并不要继续尝试破坏性操作。
```

## 12. 插件更新场景演示

这个场景用于测试人员收到新版 Android Use 插件包后，完成更新和自检。

### 12.1 场景背景

维护者发来新包：

```text
android-use-plugins.zip
```

本次更新说明：

```text
优化 WebView 输入速度，修复无线重连后 scrcpy 窗口没有复用的问题。
```

目标：

- 更新本机 Android Use 插件；
- 跑健康检查；
- 重启 Codex；
- 验证设备可连接；
- 验证 WebView 输入、截图和 scrcpy 基本可用；
- 留下更新记录。

### 12.2 给 Codex 的完整提示词

```text
我收到新版 android-use-plugins.zip，请帮我更新本机 Android Use 插件。

要求：
1. 解压新版 zip。
2. 进入解压后的插件目录执行 ./install.sh。
3. 执行 ./doctor.sh。
4. 不删除我的测试资料、截图、录屏、账号配置和 Android 设备授权。
5. 如果 doctor 失败，停止并说明失败项，不要继续假装更新成功。
6. 更新完成后提醒我重启 Codex。

重启 Codex 后，我会再让你做设备验证。
```

Codex 完成后，测试人员手动重启 Codex。

### 12.3 重启后的验证提示词

```text
[@Android] 请做 Android Use 更新后自检：

1. 检查依赖。
2. 列出设备，包含型号和屏幕信息。
3. 唤醒设备并解锁。
4. 打开 scrcpy 镜像窗口。
5. 观察当前屏幕，优先 WebView。
6. 截图并保存。
7. 如果当前页面是可调试 WebView，执行 document.title 和 location.href。

请输出：
- 设备是否可用；
- scrcpy 是否启动或复用；
- 截图保存位置；
- WebView 是否可识别；
- 是否可以开始正式测试。
```

### 12.4 WebView 输入验证

如果本次更新涉及 WebView 输入或点击，使用一个安全的测试输入框页面验证：

```text
[@Android] 请验证 Android Use 更新后的 WebView 输入能力：

约束：
1. 只在测试页面或测试账号内操作。
2. 不提交正式表单。
3. 输入文本使用 android_use_update_check_0612。

步骤：
1. 进入一个可输入文本的测试页面。
2. 观察当前页面，优先 WebView DOM。
3. 点击测试输入框。
4. 清空后输入 android_use_update_check_0612。
5. 截图并读取当前输入框值。

请说明：
- 是否走到 WebView 快路径；
- 输入框最终值；
- 截图或 AppShot 路径；
- 如失败，失败发生在哪一步。
```

## 13. 待测 App 更新场景演示

这个场景用于测试人员安装新的 App 测试包后，用 Android Use 完成冒烟验证。

### 13.1 场景背景

研发提供了新的测试包：

```text
app-test-2.8.1.apk
```

目标：

- 安装测试包；
- 确认版本号；
- 打开 App；
- 执行核心冒烟路径；
- 采集证据。

安装 APK 属于会改变设备状态的动作，需要测试人员明确授权。Android Use 本身负责设备识别、打开 App、截图、控件树、录屏和验证；APK 安装通常由 Codex 在本机执行 `adb install`，或使用团队已有安装脚本完成。

### 13.2 给 Codex 的完整提示词

```text
我授权你在当前测试设备上通过本机 adb 安装测试 APK：/path/to/app-test-2.8.1.apk。

要求：
1. 安装前先列出设备并确认只控制一台测试设备。
2. 通过 adb install 安装 APK；如果需要覆盖安装，请先说明将使用的参数。
3. 安装失败就停止并返回 adb 错误，不要继续验证。
4. 安装后用 Android Use 读取包名和版本号。
5. 打开 App。
6. 截图首页。
7. 不登录真实账号，不点击支付、删除、发送、提交订单等高风险按钮。

请输出安装结果、版本号、首页截图路径和是否可以开始冒烟测试。
```

如果团队不允许插件安装 APK，可以让测试人员手动安装，然后只做验证：

```text
[@Android] 我已经手动安装新测试包。请读取当前待测 App 的包名、版本号、前台 Activity，打开 App 首页并生成 AppShot。
```

### 13.3 冒烟验证提示词

```text
[@Android] 请对新测试包做一次冒烟验证：

范围：
1. 启动 App。
2. 首页加载。
3. 登录页可打开。
4. 测试账号可输入，但不要提交验证码。
5. 设置页可打开。

证据：
- 每个关键页面截图；
- 最后生成一次 AppShot；
- 如果出现卡死、白屏、闪退，立即停止并保留当前屏幕。

请输出通过项、失败项、证据路径和当前 App 版本。
```

## 14. 常见故障和处理

### 14.1 Codex 看不到 Android 插件

处理：

```bash
cd ~/plugins/android-use-plugins
./install.sh
./doctor.sh
```

然后重启 Codex。

如果仍然看不到，让 Codex 检查：

```text
请检查 ~/marketplace.json、~/.agents/plugins/marketplace.json 和 ~/.codex/config.toml 里是否包含 android-use-plugins。
```

### 14.2 设备 unauthorized

原因通常是设备没有授权电脑。

处理：

1. 拔掉 USB；
2. 重新插入；
3. 看设备是否弹出 USB 调试授权；
4. 选择「允许」；
5. 再让 Codex 列出设备。

### 14.3 设备列表为空

检查：

- USB 线是否支持数据传输；
- 设备是否开启 USB 调试；
- USB 连接方式是否为文件传输；
- 是否在弹窗里允许调试；
- 是否需要重启 adb。

可让 Codex 执行：

```text
[@Android] 检查依赖并解释为什么没有可用设备。
```

### 14.4 scrcpy 没窗口

先确认是否真的需要实时镜像。只要证据的话，时间线或截图也够用：

```text
[@Android] 打开 Android 操作时间线。
[@Android] 截图。
```

需要实时镜像：

```text
[@Android] 打开或复用 scrcpy 镜像窗口。
```

如果返回 already-running，通常表示已有窗口可复用，不是失败。

### 14.5 页面点不到文字

按顺序处理：

1. 先观察页面并带截图；
2. 如果是 H5 页面，列出 WebView 页面；
3. 尝试点击准确文字；
4. 尝试点击包含文字；
5. 最后才根据截图点坐标。

提示词：

```text
[@Android] 当前页面点不到“登录”。请先观察控件树和 WebView DOM，说明能识别到哪些可点击元素，不要直接猜坐标。
```

### 14.6 输入不稳定

处理：

- 对 WebView 页面，先确认是否能识别 WebView；
- 对中文或长文本，优先让插件使用输入法快路径；
- 对验证码、密码等敏感输入，按团队安全规则处理；
- 输入后截图或读取输入框值，不要只看工具返回成功。

提示词：

```text
[@Android] 清空当前输入框，输入 test_input_0612，然后验证输入框最终值并截图。
```

### 14.7 无线调试连不上

检查：

- 电脑和设备是否在同一 Wi-Fi；
- 设备无线调试是否仍然开启；
- IP 是否变化；
- 是否需要重新配对；
- 是否可先用 USB 恢复连接。

提示词：

```text
[@Android] 尝试无线重连。如果失败，请说明是 mDNS 未发现、adb connect 失败、设备未授权，还是没有保存过无线配置。
```

## 16. 最近一个月真实使用场景

这一章来自最近一个月的实际使用方式，适合测试人员直接照着做。核心原则是：先拿真实工单和设备证据，再判断是否能复现；不能复现就明确写不能复现，不要把静态代码线索写成已验证 Bug。

### 16.1 客服工单反馈日常排查

最近的学生端排查经常从「客服工单反馈」里拉取前一天的工单，再筛出学生端、Android、WebView、输入法、音频、做题、背单词等相关问题。Android Use 在这里主要负责真实设备验证和证据采集，不负责替代工单源。

推荐流程：

1. 先从真实工单入口读取数据，不要凭记忆列问题。
2. 按时间窗口筛选，例如 Asia/Shanghai 前一天。
3. 保留被纳入和被排除的工单，排除项要写原因。
4. 对 Android/WebView/输入法类问题，先确认设备是否可用。
5. 能进页面就生成截图或 AppShot；进不了页面就写清楚缺什么。
6. 结果归档到 `test-results/<date>/student-ticket-triage/`。

建议报告固定三段：

```text
工单总览
排查与修复结果
今日重点
```

建议让 Codex 这样执行：

```text
请按最近一次工单排查标准处理今天的学生端 Android/WebView 工单：

要求：
1. 优先从真实工单入口读取数据。
2. 只筛学生端相关工单，列出纳入和排除原因。
3. 对 Android/WebView/输入法类问题，先检查设备连接和 App 版本。
4. 能复现的先复现，不能跳过复现直接改代码。
5. 没有设备、账号、课程、录屏或后端数据时，明确标注未复现或待补证据。
6. 报告写入 test-results/<date>/student-ticket-triage/report.md。
7. 输出结构固定为：工单总览 / 排查与修复结果 / 今日重点。
```

### 16.2 Android/WebView 工单复现

学生端常见形态是少量原生入口加大量 H5/WebView 页面。最近的有效做法不是一上来截图点坐标，而是先确认设备、App、前台 Activity 和 WebView。

推荐步骤：

```text
[@Android] 列出设备，包含型号和屏幕信息。
[@Android] 唤醒设备并解锁。
[@Android] 打开待测学生端 App。
[@Android] 生成当前 AppShot，包含截图、设备状态和控件树。
[@Android] 列出当前可调试 WebView 页面。
[@Android] 如果有 WebView，读取 document.title 和 location.href。
```

结论写法要区分：

- `已复现`：在正确设备、正确 App 版本、正确账号和正确页面路径下看到了问题，并有截图、AppShot 或录屏。
- `未复现`：按工单路径操作过，但没有看到问题。
- `设备可用、未复现`：设备在线，但缺少受影响账号、课程、单词、任务、录屏或后端数据。
- `版本不匹配`：本机安装版本低于或不同于工单版本，不能代表工单版本结论。
- `只读排查/待补证据`：只拿到了静态线索或页面状态，没有完整复现链路。

实际使用里，设备显示 `HONOR ELN-W09`、序列号可见、甚至前台是 `StandardBrowserActivity`，只能说明具备学生端 WebView 验证能力，不能直接说明某个工单已复现。还需要对应账号、页面路径、业务数据和截图/录屏。

### 16.3 版本不匹配时不要硬判定

最近多次遇到工单版本和本机安装版本不一致。例如工单反馈版本是 `1.14.0199`，设备上实际安装的是 `1.13.0102`。这种情况下，即使页面能打开，也不能用旧版本结果覆盖新版本结论。

推荐提示词：

```text
[@Android] 请读取当前待测 App 的包名、versionName、versionCode 和前台 Activity。
如果版本低于工单版本，请停止复现并把结论写成“版本不匹配/设备可用、未复现”，不要继续推断新版本表现。
```

报告写法：

```text
结论：版本不匹配，设备可用但未复现。
证据：当前设备 versionName=1.13.0102，工单版本=1.14.0199。
处理：需要安装对应测试包或拿到同版本设备后再复现。
```

### 16.4 原生问题和 H5 问题要先分界

很多学生端问题会落在 Android 原生、WebView H5、后端数据、课程配置之间。Android Use 的价值是先把边界打清楚，而不是把所有问题都归到当前 Web 仓库。

最近遇到过这些判断模式：

- 工单指向 `MarkableTextView.kt`、播放器速度、音频焦点等 Kotlin/原生组件，但当前 Web 仓库里没有对应实现，应先标注 Android 原生侧或 native-runtime 归属，不要强行开 H5 修复。
- H5 页面可以通过 `location.href`、页面标题、DOM 文本和 AppShot 确认；如果只有原生容器 Activity，没有 H5 URL 和页面证据，还不能说 H5 复现。
- 代码搜索到 `useTimeSpent`、`AvatarWithFrame`、`question-store.tsx` 这类线索，只能作为静态缩小范围，不能替代复现。
- 课程、任务、班级、历史数据不一致时，先写数据/配置调查，不要直接写前端 Bug。

建议提示词：

```text
[@Android] 请帮我判断这个工单更像 H5、Android 原生、后端数据还是配置问题。

请只读排查：
1. 当前设备和 App 版本；
2. 前台 Activity；
3. WebView title 和 location.href；
4. 当前页面 AppShot；
5. 是否有可点击/可输入的 H5 DOM；
6. 如果需要代码线索，只列为“静态线索”，不要写成已复现。

最后按 H5 / Android 原生 / 后端数据 / 配置 / 证据不足 给出归类。
```

### 16.5 静态代码线索不能替代复现

有些工单能在代码里找到可疑点，但仍然不能直接修。例如：

- 某个按钮是否可提交，代码里能看到 `canSubmit` 需要有效答案；如果工单线程里已经有人确认用户该题完成订正，就更像未复现或状态已变化。
- 某个交互弹窗或课程组件有静态线索，但缺少课程录屏、账号和具体课节，只能写 `需进一步验证`。
- 某个音频/播放器问题在 Web 代码里找不到原生服务或 AudioFocus 实现，应停止在 Web 侧硬修。

推荐报告措辞：

```text
当前只拿到静态代码线索，未形成完整复现链路。
结论：需进一步验证。
缺口：缺少受影响账号 / 课程 / 录屏 / 对应版本安装包。
下一步：补齐缺口后用 Android Use 复现，并保存 AppShot 或 MP4。
```

### 16.6 重跑工单时先判断有没有新回复

同一批工单重复排查时，不要每次都重新展开所有分析。最近更高效的做法是保存原始消息、线程 JSON 和报告；重跑时先比较 thread JSON 是否变化。

建议归档结构：

```text
test-results/<date>/student-ticket-triage/
├── raw/
│   ├── messages.json
│   ├── chat-search.json
│   └── threads/<ticket-id>.json
├── android/
│   ├── appshot-*.json
│   └── screenshot-*.png
└── report.md
```

如果重跑发现线程没有新增回复，报告可以直接写：

```text
与上次原始线程数据一致，未发现新增回复。
本次不重新打开修复工作流，保留上次结论。
```

### 16.7 工单排查中的 Android Use 最小证据包

一次学生端 Android/WebView 工单排查，建议至少保留这些证据：

- `android_list_devices` 结果：设备型号、序列号、Android 版本；
- App 包名和版本号；
- 前台 Activity；
- 截图或 AppShot；
- WebView title 和 URL，若页面可调试；
- 工单线程原文或摘要；
- 结论状态：已复现 / 未复现 / 设备可用未复现 / 版本不匹配 / 待补证据 / 原生侧归属 / 数据配置问题；
- 下一步动作和缺口。

可以直接使用下面的提示词：

```text
[@Android] 请为当前工单生成最小证据包：

1. 设备型号、Android 版本、序列号；
2. 当前 App 包名、versionName、versionCode；
3. 前台 Activity；
4. 当前截图和 AppShot；
5. 如果是 WebView，读取 title 和 location.href；
6. 判断是否满足复现前提；
7. 如果不满足，请输出缺口，不要继续猜测。

结论只能从以下状态中选择：
已复现 / 未复现 / 设备可用未复现 / 版本不匹配 / 待补证据 / 原生侧归属 / 数据配置问题。
```

### 16.8 什么时候可以进入修复流程

测试人员排查工单时，不要因为看到可疑代码就直接让 Codex 改代码。进入修复至少需要满足：

1. 工单来源明确；
2. 设备或环境可用；
3. App 版本匹配，或确认问题与版本无关；
4. 有账号、课程、任务、单词、页面路径等业务前提；
5. 有截图、AppShot 或录屏；
6. 能判断是 H5/Web 侧、Android 原生侧、后端数据还是配置问题；
7. 如果要改代码，先使用独立 worktree，不要在主仓库直接改。

如果缺少其中任意关键项，报告应停在排查结论，不要写成修复完成。

## 17. 推荐测试习惯

- 每轮测试开始先列设备，避免控制错设备；
- 关键页面用 AppShot，不只截屏；
- 动态问题用 MP4，静态页面用截图或 AppShot；
- 重复回归路径录 recipe；
- WebView 页面先查 DOM，再考虑截图坐标；
- 不确定页面状态时先观察，不要猜；
- 高风险按钮明确禁止或单独确认；
- 更新插件后必须跑 `./doctor.sh` 并重启 Codex；
- 更新 App 后必须记录包名、版本号和首页证据；
- 结论里区分「通过」「未通过」「未复现」「阻塞」，不要把设备不可用写成测试通过。
